# AI 开发规则（Rules）

你是该项目的资深全栈工程师，所有代码和文档修改都必须遵守以下规则。

## 核心原则

1. 优先保证简单、清晰、可维护
2. 不随意引入新的框架或大型依赖
3. 遵守当前项目的单仓全栈架构
4. 前后端职责边界必须清晰
5. 优先复用已有逻辑、组件和工具函数
6. 先解决真实问题，再做抽象
7. 先做 MVP，再考虑扩展
8. 默认不包含发布、部署、CI/CD

## 当前项目架构约束

前端：

- 交付形态为微信小程序
- 技术栈以 `Vue 3 + uni-app + TypeScript` 为主
- 状态管理使用 `Pinia`
- 网络请求统一走 `Axios` 封装

后端：

- 使用 `Node.js + Express + TypeScript`
- 数据访问统一走 `Prisma`
- 数据库默认使用 `MySQL`

协作约束：

- 商品价格、库存、订单金额等核心业务逻辑必须在后端
- 前端页面不直接处理复杂业务规则
- 前端不直接拼接接口细节，统一从 `api` 层发起请求
- 接口返回结构保持统一

## 代码规范

- 前后端优先使用 TypeScript
- 不滥用 `any`，必要时说明原因
- 函数保持短小清晰
- 避免深层嵌套逻辑
- 变量命名必须语义化
- 接口、DTO、状态类型尽量显式声明
- 复杂逻辑优先拆到 service、hooks 或 utils

## 工程化规范

- 必须兼容 ESLint 和 Prettier 规则
- 提交前优先执行最小必要检查
- 新增依赖前先判断是否已有现成能力
- 修改时优先保持目录结构稳定，不做无意义重构

## 修改前必须做的事

1. 说明将要修改哪些文件
2. 解释为什么需要修改
3. 说明可能带来的风险
4. 确认是否已有可复用逻辑

## 修改完成后必须做的事

1. 列出所有修改文件
2. 总结实现逻辑
3. 给出验证方法
4. 明确 review 结论
5. 如果存在已知限制，必须明确说明

## 默认不做的事

- 不默认新增管理后台
- 不默认引入微服务
- 不默认设计复杂营销系统
- 不默认补部署流水线
